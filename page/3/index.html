1、什么是Java？

Java是一门面向对象的编程语言，它有功能强大和简单易用两个特征。它继承了C++语言面向对象的技术核心，摒弃了C++里的多继承、指针，增加了垃圾回收机制，释放掉不被使用的内存空间，解决了管理内存空间的烦恼。

1，HashMap和HashTable有何不同？（会）

  • HashTable继承的是Directory类，HashMap是Map的接口的一个实现
  • HasTable是线程安全的，方法是同步的；HashMap是线程不安全，不同步，效率比HashTable高
  • HashMap能够存储null键或null值，HashTable不允许
  • HashMap的iterator的迭代器是fail-fast的，而HashTable的enumerator迭代器不是fail-fast的

相同点：

  • 两者维护插入的顺序，可以根据插入顺序来获取元素。

同步性：Vector线程安全，方法之间是线程同步的；ArrayList线程不安全，方法之间不同步。如果单线程最好使用ArrayList，多线程最好使用Vector

数据增长：当add的元素的个数超过了初始的容量大小，就会进行扩增，vector扩增为原来的2倍，ArrayList扩增为原来的1.5倍。

设置增长空间：Vector有可以设置增长的空间大小的方法，ArrayList没有这个方法。

4， 队列和栈是什么，列出它们的区别？（会）

栈和队列都是预存数据的数据结构。

进出策略不同：栈堆元素是先进后出(LIFO)进行检索，队列对元素进行先进先出(FIFO)的检索，双端队列允许从两端检索元素元素。

类型不同：栈是一个扩展自Vector的类，而队列是一个接口，它的实现类在java并发包中。

5， Comparable和Comparator接口是什么？（会）

当要实现Array或Collection的排序方法时，需要在自定义类里实现Java提供Comparable接口。

Comparable有compareTo方法，它被排序方法所使用，调用这个方法的对象比传递的对象参数小、相等、更大时候，返回一个负整数、0、正整数。

Comparator有compare方法，第一个参数比第二个参数小、相等或者更大，分布返回负整数、0、正整数。

7 ，多线程有什么用？（会）

  • 返回多核CPU的优势，可以让多段逻辑同时工作，多线程，充分利用CPU
  • 防止阻塞。多条线程同时执行，即使有一条线程的代码执行的时候被堵塞，也不会影响其他的线程上任务的执行。
  • 便于建模。单线程编程的程序模型建立比较麻烦，多线程可以分成几个小任务，分布建立程序模型，并通过多线程分别运行这几个任务。

8， 多线程和单线程的区别和联系？（Again）

  • 单核CPU中，CPU分为很小的时间片，每一时刻只能有一个线程在执行，是一种微观上轮流占用CPU的机制
  • 多线程会存在线程上下文切换，会导致程序执行速度变慢。一个拥有两个线程的进行执行所需要的时间要多于拥有一个线程的进程执行两次所需要的时间。

9， 简述线程、程序、进程的基本概念。以及他们之间关系是什么?

线程

线程是一个比进程更小的执行单位。一个进程可以产生多个线程。与

进程不同的是同类的多个线程共享同一块内存空间和一组系统资源，所以系统在产生一个线程，或是在

各个线程之间作切换工作时，负担要比进程小得多，也正因为如此，线程也被称为轻量级进程。

程序

是含有指令和数据的文件，被存储在磁盘或其他的数据存储设备中，也就是说程序是静态的代码。

进程

是程序的一次执行过程，是系统运行程序的基本单位，因此进程是动态的。系统运行一个程序即是一个

进程从创建，运行到消亡的过程。简单来说，一个进程就是一个执行中的程序，它在计算机中一个指令

接着一个指令地执行着，同时，每个进程还占有某些系统资源换句话说，当程序在执行时，将会被操作系统载入内存中。 线程是进程划分成的更小的运行单位。

线程和进程最大的不同在于基本上各进程是独立的，而各线程则不一定，因为同一进程

中的线程极有可能会相互影响。从另一角度来说，进程属于操作系统的范畴，主要是同一段时间内，可

以同时执行一个以上的程序，而线程则是在同一程序内几乎同时执行一个以上的程序段。

5、什么是线程安全

线程安全就是说多线程访问同一代码，不会产生不确定的结果。

多线程环境中，非共享的数据是线程安全的，但是多数情况是共享数据的。

线程安全一般都涉及到synchronized，即一段代码只能有一个线程来操作。

如果代码所在的进程有多个线程同时执行，这些线程又有可能同时运行这段代码，如果每次运行结果和单线程的运行结果一样，其他的变量值也跟预期一样，那么就线程安全。

6、线程的状态转换（会）

new Thread()就是新建状态

就绪状态是新建状态之后调用了start方法，等待获取CPU的使用权

运行状态是就绪状态的线程获取了CPU，执行程序代码

阻塞状态时线程因为某种原因放弃了CPU的使用权，暂时停止运行。

阻塞状态分为三种：等待阻塞、同步阻塞、其他阻塞

死亡状态时线程执行完毕或者因异常退出了run方法，结束了生命周期。

9、用户线程和守护线程有什么区别？（会）

守护线程都是为JVM中所有非守护线程的运行提供便利服务： 只要当前JVM实例中尚存在任何一个非守护线程没有结束，守护线程就全部工作；只有当最后一个非守护线程结束时，守护线程随着JVM一同结束工作。

User和Daemon两者几乎没有区别，唯一的不同之处就在于虚拟机的离开：如果 用户线程已经全部退出运行了，只剩下守护线程存在了，虚拟机也就退出了。因为没有了被守护者，Daemon也就没有工作可做了，也就没有继续运行程序的必要了。

1、sleep方法和wait方法有什么区别

  • 所属类不同：sleep属于线程类中的，wait属于Object类
  • 作用范围：wait方法作用于同步控制方法或代码块里，sleep作用于任何地方
  • 作用不同：sleep导致程序暂停，不会释放对象锁，监控状态仍然保持着，时间到了会自动恢复到可运行状态；wait方法会放弃对象锁，进入等待锁定池，只有调用notify才会进入可运行状态

2、线程的sleep()方法和yield()方法有什么区别？

  • sleep方法给其他线程运行机会的时候不考虑线程的优先级，yield方法只给同级或者更高优先级线程运行机会
  • 线程执行sleep()方法后转入阻塞状态,所以,执行sleep()方法的线程在指定的时间内不会被执行,而yield方法只是使当前线程重新回到可执行状态,所以执行yield()方法的线程可能在进入可执行状态后马上又被执行
  • sleep()方法声明抛出InterruptedException,而yield()方法没有声明任何异常
  • sleep比yield(跟操作系统相关)有更好的可移植性

比如执行顺序是T1、T2、T3，那就再T3使用Join调用T2、再T2中使用Join调用T1

都是Object对象用于通知处在等待该对象的线程的方法，都是在同步代码块里被调用

5、什么是线程池？ 为什么要使用它？

概念：为了避免创建线程所花费高昂的资源和时间，在程序启动的时候就创建若干线程来响应处理，被称为线程池。所以线程池是一种多线程处理形式，处理过程中将任务添加到队列，创建线程后自动启动这些任务。（个人理解：事先创建若干个而执行的线程放入一个池中，需要的时候取出来（不需要自行创建），不需要的时候放回池中）

为什么要使用（优点）：减少创建与销毁线程所带来的开销，提高响速度，提高线程的可管理性

8、Java中的死锁

死锁是两个或两个以上的线程在执行过程中，由于竞争资源或由于彼此通信而造成的现象，若无外力作用，他们都无法推进下去。死锁的四个必要条件：

  • 互斥条件：某一资源只能被某一个进程所请求
  • 请求和保持条件
  • 不剥夺条件
  • 环路等待条件

9、如何避免死锁和检测

预防死锁：

  • 破坏互斥条件
  • 破坏请求和保持条件
  • 破坏不剥夺条件
  • 破坏循环等待条件

10、什么是乐观锁和悲观锁

乐观锁：在并发操作时候，乐观锁在操作数据时非常乐观，认为别人不会同时修改数据。因此乐观锁不会上锁，只是在执行更新的时候判断一下在此期间别人是否修改了数据：如果别人修改了数据则放弃操作，否则执行操作。
悲观锁：悲观锁在操作数据时比较悲观，认为别人会同时修改数据。因此操作数据时直接把数据锁住，直到操作完成后才会释放锁；上锁期间其他人不能修改数据。

3**、**程序计数器（Program Counter Register）,程序计数器（Program Counter Register）是一块较小的内存空间，它的作用可以看做是当前线程所执行的字节码的行号指示器。

4**、**JVM栈（JVM Stacks）,它的生命周期与线程相同。用于存储局部变量表、操作栈、动态链接、方法出口等信息。

5**、**本地方法栈（Native Method Stacks）,本地方法栈（Native Method Stacks）与虚拟机栈所发挥的作用是非常相似的，其区别不过是虚拟机栈为虚拟机执行Java方法（也就是字节码）服务，而本地方法栈则是为虚拟机使用到的Native方法服务。

简述 Java 垃圾回收机制。

在 JVM 中，有一个垃圾回收线程，它是低优先级的，在正常情况下是不会执行的，只有在虚拟机空闲或者当前堆内存不足时，才会触发执行，扫面那些没有被任何引用的对象，并将它们添加到要回收的集合中，进行回收

垃圾收集算法

GC最基础的算法有三种： 标记 -清除算法、复制算法、标记-压缩算法，我们常用的垃圾回收器一般都采用分代收集算法。

1**、标记** -**清除：首先标记出所有需要回收的对象，在标记完成后统一回收掉所有被标记的对象。

2**、复制算法**，“复制”（Copying）的收集算法，它将可用内存按容量划分为大小相等的两块，每次只使用其中的一块。当这一块的内存用完了，就将还存活着的对象复制到另外一块上面，然后再把已使用过的内存空间一次清理掉。

3**、标记**-**压缩算法**，标记过程仍然与“标记-清除”算法一样，但后续步骤不是直接对可回收对象进行清理，而是让所有存活的对象都向一端移动，清理7端边界以外的内存

4****、分代收集算法****，把Java堆分为新生代和老年代，这样就可以根据各个年代的特点采用最适当的收集算法。

TCP与UDP区别：

1**、**TCP是面向连接的协议，发送数据前要先建立连接，TCP提供可靠的服务，也就是说，通过TCP连接

传输的数据不会丢失，没有重复，并且按顺序到达；

2**、**UDP是无连接的协议，发送数据前不需要建立连接，是没有可靠性；

3**、**TCP通信类似于于要打个电话，接通了，确认身份后，才开始进行通行；

4**、**UDP通信类似于学校广播，靠着广播播报直接进行通信。

5**、**TCP只支持点对点通信，UDP支持一对一、一对多、多对一、多对多；

6**、**TCP是面向字节流的，发送数据时以字节为单位；UDP是面向报文的，一个报文只能一次发完。

7**、**TCP首部开销（20字节）比UDP首部开销（

8字节）要大

8**、**UDP 的主机不需要维持复杂的连接状态表

TCP和UDP的应用场景：

对实时性要求比较高的情况使用UDP，比如游戏，通信，实时直播；其它大部分情况下，HTTP都是用TCP，因为要求传输的内容可靠，不出现丢失的情况

10、唯一索引比普通索引块吗？为什么

2、更新时，这个情况就比较复杂了. 普通索引将记录放到 change buffer 中语句就执行完毕了. 而对唯

1. 锁机制与InnoDB锁算法

表级锁和行级锁对比：

  • 表级锁：MySQL中锁定粒度最大的一种锁，对当前操作的整张表加锁，开销小，不会出现死锁，触发所冲突的频率最高，并发度最低
  • 行级锁：MySQL中锁定粒度最小的一种锁，值对当前操作的行进行加锁，开销最大，它能减少数据库操作的冲突。加锁慢，会出现死锁，并发度高

InnoDB的锁算法：

  • Record lock：单个记录上的锁
  • Gap lock：间隙锁，锁定一个范围，不包括记录本身
  • Next-key lock：record+gap锁定一个范围，包含记录本身

  • no：Spring框架的默认设置，该模式下自动装配式关闭的
  • byName：该选择开源根据bean名称设置依赖关系，根据bean名称查找bean，找到就装配，找不到报错。
  • byType：该选择开源根据bean类型设置依赖关系。根据bean的类型自动查询，找到了装配，找不到报错。
  • constrictor：构造器的自动装配和byType模式类型，但是仅仅适用于有构造器相同参数的bean，如果没有找到就抛出异常
  • autodetect：该模式自动探测使用构造器自动装配或byTyep自动装配，首先找带参的构造器，找到了自动装配，没找到容器会自动选择byType的自动装配方式。

3、SpringMvc执行流程和原理SpringMVC流程（已）

  • 用户发起请求到中央控制器
  • 中央控制器调用处理器映射器
  • 处理器映射器找到具体的处理器，生成处理器对象以及处理器拦截器，再一起返回给中央控制器
  • 中央控制器调用处理器适配器
  • 处理器适配器调用具体的处理器
  • 执行完毕返回ModelAndView对象，处理器适配器将对象返回给中央控制器，中央控制器将对象传给视图解析器，视图解析器解析后返回视图
  • 中央控制器根据View进行渲染，响应给用户

5、@Component、@Controller、@Repository、@Service的区别？

@Component：式Spring管理组建的通用构造型，将类标记为bean

@Controller：将类标记为Spring Web MVC控制器，并自动导入到IOC容器中

@Service：注解式组件注解的特化，比如服务层使用这个注解而不俗Component

@Repository：注解的特化，比如控制层使用这个注解，将dao导入ioc容器

6、@Autowire和@Resource区别？

  • 注解来源：前者是Spring注解，后者是JDK注解
  • 装配方式：前者优先按类型，后者优先按名称
  • 属性：前者是required，后者是name、type
  • 作用范围：前者是字段、setter方法、构造器，后者是字段、settter方法

7、Spring的IOC理解？（已）

  • IOC就是控制反转，是指创建对象的控制权的转移，以前创建对象的主动权和时机是由自己把控的，
    而现在这种权力转移到Spring容器中，并由容器根据配置文件去创建实例和管理各个实例之间的依赖关
    系，对象与对象之间松散耦合，也利于功能的复用。DI依赖注入，和控制反转是同一个概念的不同角度
    的描述，即 应用程序在运行时依赖IoC容器来动态注入对象需要的外部资源。
  • 2、最直观的表达就是，IOC让对象的创建不用去new了，可以由spring自动生产，使用java的反射机制，根据配置文件在运行时动态的去创建对象以及管理对象，并调用对象的方法的。
  • 3、Spring的IOC有三种注入方式 ：构造器注入、setter方法注入、根据注解注入。
    IoC让相互协作的组件保持松散的耦合，而AOP编程允许你把遍布于应用各层的功能分离出来形成可重用
    的功能组件

8、列举IOC的一些好处？（已）

  • 它将最小化应用程序中的代码量。
  • 它将使应用程序易于测试，因为它不需要单元测试用例中的任何单例或 JNDI 查找机制。
  • 它以最小的影响和最少的侵入机制促进松耦合。
  • 它支持即时的实例化和延迟加载服务

9、spring通知有哪些类型？（已）

  • before：在一个方法执行前被调用。
  • after: 在方法执行之后调用的通知，无论方法执行是否成功。
  • after-returning: 返回后通知，仅当方法成功完成后执行的通知。
  • after-throwing: 抛出异常时执行的通知。
  • around: 在方法执行之前和之后调用的通知

1、Spring Bean有哪些作用域，它们之间有什么区别？

  • singleton：这种bean范围是默认的，这种范围确保不管接受到多少个请求，每个容器只有一个bean的实例
  • prototype：为每个bean请求提供一个实例
  • request：在请求bean范围内会每一个来自客户端的网络请求创建一个实例，请求完成以后，会被垃圾回收器回收
  • session：每个session有一个bean的实例，过期bean会失效。
  • global-session：如果想让所有的prolet工用全局的存储变量的话，那么全局变量需要存储在global-session中。

2、Spring框架有哪些自动装配模式，有什么区别？

  • no：是Spring框架的默认设置，该设置下自动装配式关闭的
  • byName：根据bean的名称设置依赖关系，查找匹配，如果找到就装配，没找到报错
  • byType：根据bean类型设置依赖关系，查找匹配，如果找到就装配，没找到就报错
  • constructor：构造器的自动装配和byType模式，但仅适用于与构造器相同参数的bean，没有匹配到就抛出异常
  • autodetect：该模式自动探测使用构造自动装配和byType自动装配。

1、什么是MyBatis？

  • MyBatis是一个半ORM框架，内部封装了JDBC，开发时只需要关注SQSSL语句本身，不需要去处理加载驱动、创建链接等过程。
  • MyBatis可以使用XML或注解来配置和映射原生信息，避免了几乎所有的JDBC代码和手动设置参数以及获取结果集
  • 通过xml文件或注解的方式将要执行的各种statement配置起来，并生成最终执行的sql语句

2、MyBatis的优点？

  • 不会对程序或数据库造成任何影响
  • 降低sql与程序的耦合，便于统一管理；支持编写动态sql语句，并可重用
  • 减少了代码量
  • 能与各种数据库兼容，与Spring很好的集成
  • 提供映射标签，支持对象与数据库ORM字段关系映射；提供对象关系映射标签，支持对象关系组件维护

3、MyBatis与Hibernate区别？

相同：都是对JDBC的封装，都是持久层的框架，都是用于dao层的开发

不同：

  • 映射关系不同，MyBatis是一个半rom框架，配置Java对象与sql语句执行结果的对应关系，多表关联关系配置简单；Hibernate是一个全ORM的框架，配置Java对象与数据库表的对应关系，多表关联关系配置复杂
  • sql优化和移植性不同，需要MyBatis需要手动编写SQL，支持动态SQL，直接使用SQL操作数据库，不支持数据库无关性，但sql语句优化更容易。

1、MyBatis是如何进行分页的？分页插件的原理是什么？

  • Java Bean传参法：参数名与成员属性名保持一致

3、MyBatis是否支持延迟加载？如果支持，它的实现原理是什么？

  • 支持。MyBatis的核心配置文件可以配置lazyLodingEnabled，来决定是否启用延迟加载。
  • 它的原理是，使用CGLIB创建目标对象的代理对象，当调用目标方法时，进入拦截器方法，比如调用a.getB().getName()，拦截器invoke()方法发现a.getB()是null值，那么就会单独发送事先保存好的查询关联B对象的sql，把B查询上来，然后调用a.setB(b)，于是a的对象b属性就有值了，接着完成a.getB().getName()方法的调用。这就是延迟加载的基本原理。

4、MyBatis动态sql有什么用？执行原理？有哪些动态sql？

Mybatis动态sql可以在Xml映射文件内，以标签的形式编写动态sql，执行原理是根据表达式的值 完成逻辑判断并动态拼接sql的功能。Mybatis提供了9种动态sql标签： trim|where|set|foreach|if|choose|when|otherwise|bind 。其执行原理为，使用 OGNL 从 sql 参数对象中计算表达式的值，根据表达式的值动态拼接 sql，以此来完成动态 sql 的功能。

5、MyBatis的一级、二级缓存？

一级缓存：基于PerpetualCache中的HashMap本地缓存，其存储作用域为Session，当Session flush或close之后，该Session中的所有Cache就将清空，默认打开一级缓存。

二级缓存与一级缓存机制相同，不同于其存储域为Mapper，并且可以自定义存储源，为Ehcache。默认不打开二级缓存，要开启二级缓存，使用二级缓存属性类需要实现Serializable序列化接口，可在它的映射文件中配置<cache>

对于缓存数据更新机制，当一个作用域进行了CUD操作后，默认作用域下所有select中的缓存将被clear
